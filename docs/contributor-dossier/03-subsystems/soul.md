# Subsystem: Soul

The personality system. Optional. Off by default.

---

## What it is

`soul/` is a directory of Markdown files that Aeon reads before every skill execution to embody a consistent voice. Without `soul/`, Aeon writes in a neutral / generic tone. With a populated `soul/`, every notification, article, tweet, and reply sounds like the operator.

## File hierarchy

The order matters; later files refine earlier ones:

| File | Purpose |
|---|---|
| `soul/SOUL.md` | Identity, worldview, opinions, background. *Who* the agent is. |
| `soul/STYLE.md` | Writing style: sentence structure, vocabulary, punctuation, anti-patterns. *How* it writes. |
| `soul/examples/` | Calibration material — sample tweets, conversations, "bad outputs" annotated with why they're bad. *Reference cases* for style. |
| `soul/data/` | Raw source material — articles, influences. *Grounding*, not copy-paste. |

Read order is encoded in `CLAUDE.md` § Voice:

> Read in this order: `soul/SOUL.md` → `soul/STYLE.md` → `soul/examples/` → `soul/data/`.

## How it propagates

`CLAUDE.md` is loaded into the context of every skill run by Claude Code automatically. The Voice section at the top of `CLAUDE.md` tells every skill: *if `soul/` files exist, read them before writing any output.* That's the entire propagation mechanism — no per-skill wiring, no soul-aware runtime.

This is elegant and fragile in the same breath. Elegant because adding `soul/` files is the only step needed for personality to flow into 156 skills. Fragile because a skill that ignores `CLAUDE.md` (or runs with `CLAUDE.md` truncated by context-window limits) writes in a neutral voice — and there is no test that catches it.

## The opt-in story

By default, the upstream `aeon` repo ships with **empty** `soul/SOUL.md` and `STYLE.md` placeholders, plus example structure. To activate:

1. Fork [`soul.md`](https://github.com/aaronjmars/soul.md) (the recommended starter template).
2. Fill in `SOUL.md` (identity, worldview, opinions, interests).
3. Fill in `STYLE.md` (voice, sentence patterns, vocabulary, tone).
4. Add 10–20 calibration samples in `soul/examples/good-outputs.md`.
5. Copy into your Aeon fork under `soul/`.
6. Add the Identity block to the top of your `CLAUDE.md`:
   ```markdown
   ## Identity

   Read and internalize before every task:
   - `soul/SOUL.md` — identity and worldview
   - `soul/STYLE.md` — voice and communication patterns
   - `soul/examples.md` — calibration examples

   Embody this identity in all output. Never hedge with "as an AI."
   ```

The README's quality check is worth repeating: *"soul files work when they're specific enough to be wrong."* `"I think most AI safety discourse is galaxy-brained cope"` is a useful soul claim. `"I have nuanced views on AI safety"` is not.

## What soul does *not* do

- It does **not** override skill prose. A skill that says "respond in JSON" overrides soul tone.
- It does **not** propagate into the dashboard UI — the dashboard is templated, not generated by Claude.
- It does **not** propagate into the MCP / A2A tool descriptions — those come from `skills.json` (auto-generated from `SKILL.md` frontmatter, soul-blind).
- It does **not** affect quality scoring — the Haiku scorer judges utility, not voice fit.

## Anti-patterns observed

- **Soul that is mostly disclaimers.** ("I am a balanced, thoughtful, ethical AI that…") — soul is for *taking sides*, not avoiding them. Disclaimers neutralize voice.
- **Soul that contradicts skill prose.** ("Always be concise" in soul, "Write a 3000-word article" in skill prose) — skill wins. Soul should set the *flavor*, not the *length*.
- **Soul that imitates a specific person without consent.** Soul is for *operator voice*, not impersonation. Treat impersonation as a security / ethics concern.

## Testing soul

There is no programmatic test for soul. The pragmatic check:

1. Pick a skill with high-volume output (`article`, `digest`, `morning-brief`).
2. Run it twice — once with `soul/` populated, once with `soul/` emptied.
3. If the outputs are indistinguishable, soul is not propagating (or your soul is too generic).
4. If they're distinguishable but the soul version is worse, your soul is too specific or contradicts the skill.

This is part of what the `skill-evals` skill does at the fleet level — it scores outputs against assertion criteria, some of which can include voice fit if you write them.
