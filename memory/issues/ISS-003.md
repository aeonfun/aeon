# ISS-003 — Skills write the `## Summary` blob into `.outputs/` instead of the locked artifact

## Summary

Chain skills are expected to write a downstream-consumable, locked-format artifact
to `.outputs/{skill}.md`. On some runs the skill instead writes its assistant-facing
`## Summary` block into that file. Two occurrences on 2026-05-18:

1. **narrative-tracker** — the first 2026-05-18 run wrote a `## Summary` blob into
   `.outputs/narrative-tracker.md`. Caught and fixed by a corrective re-run that
   regenerated the artifact in the v2-locked format. The re-run log noted the bug is
   "worth a SKILL.md/skill-health note if it recurs."
2. **perps-scan** — the seventh 2026-05-18 run left `.outputs/perps-scan.md`
   containing the assistant `## Summary` blob (`## Summary` / `**What I did**` /
   `**Result**` / `**Files**` / `**Follow-ups**`), not the v2 locked format with the
   full per-asset raw metric table. The blob even states the file was "overwritten
   with the v2 locked format" — which is false.

Since the pattern flagged in (1) has now recurred in (2), this is filed as a tracked
issue.

## Impact

- Low-to-medium. The Summary blob happens to carry the headline verdict and the key
  per-asset numbers, so `perps-brief` was still able to consume the perps-scan
  verdict (BSB CATALYST-BREAKOUT) and produce its brief.
- The locked-format payload (full per-asset raw metric table for perps-scan;
  v2-locked signal for narrative-tracker) is lost for any consumer that needs more
  than the headline — and for cross-day audit skills reading historical artifacts.
- Risk grows if a future consumer parses the artifact structurally rather than
  reading prose.

## Recommended fix

In the affected SKILL.md files, make the artifact-write step explicit and separate
from the assistant `## Summary`:

- State that `.outputs/{skill}.md` must contain ONLY the locked-format payload —
  never the assistant `## Summary` block.
- Have the skill write the artifact from a named, pre-composed locked-format string,
  then compose the `## Summary` separately for the chat response.

Both perps-scan and narrative-tracker should be patched; audit other chain skills
for the same dual-output ambiguity.

## Workaround

`.outputs/perps-scan.md` for 2026-05-18 currently holds the Summary blob. A
perps-scan re-run regenerates it in the correct format (as was done for
narrative-tracker today). Not done automatically by this skill — daily-ops-review is
consume-only.

## Resolution (2026-05-19)

Patched both affected SKILL.md files with an explicit "CRITICAL — artifact vs assistant Summary separation (ISS-003 guardrail)" callout right at the artifact-write step. The callout:

- States that the artifact file content is the locked-format text and only that
- Lists what the artifact is NOT (the assistant `## Summary` block, "What I did" prose, narration of the result)
- Instructs the skill to compose the locked-format payload as a string FIRST, write it, and only THEN compose the `## Summary` separately for the chat-side response
- Gives a recovery cue: "If the artifact ever begins with `## Summary` or `**What I did**`, it is wrong — overwrite"

Will observe across upcoming morning-review chain runs. If the bug recurs, the guardrail prose isn't strong enough and we'll need a structural fix (e.g. write the artifact via a templated jq command or a separate sub-skill that has no Summary in its context).

Files patched:
- skills/perps-scan/SKILL.md (step 8)
- skills/narrative-tracker/SKILL.md (step 5)
