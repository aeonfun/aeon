## What happened

Two consecutive mornings of missed cron windows on the aeon repo:

- **2026-04-30**: 07:00 + 07:30 UTC slots silently skipped. The 09:01 UTC
  catch-up tick dispatched only 08:00 (heartbeat) + 09:00 (issue-triage,
  pr-review, etc.). Logged in `memory/logs/2026-04-30.md` 08:00 heartbeat
  entry as "GHA cron tick gap 06:37 → 09:01" — flagged as transient at
  the time, not notified.
- **2026-05-01**: 07:00 + 07:30 + 08:00 UTC slots all silently skipped.
  At 08:53 UTC, `cron-state.json` shows zero dispatches with timestamps
  in 2026-05-01 — last_dispatch values across every skill are still
  ≤ 2026-04-30. This heartbeat run is firing because the operator
  invoked it manually, not because GHA scheduler triggered it.

Per `memory/MEMORY.md` directive ("if recurs 05-01, file as ISS-017"),
this issue is now formal.

## Signal

- `cron-state.json` shows no `last_dispatch` value with a 2026-05-01
  prefix as of 2026-05-01T08:53Z UTC. All scheduled skills show their
  most-recent dispatch on 2026-04-30 (or earlier).
- Affected skills with concrete miss evidence:
  - `chain:morning-brief` last_dispatch 2026-04-29T07:11:45Z (now 49h+ idle vs 24h schedule)
  - `daily-routine` last_dispatch 2026-04-29T07:11:45Z (49h+)
  - `rss-digest` / `paper-digest` / `reddit-digest` last_dispatch 2026-04-29T07:11:45Z (49h+)
  - `telegram-digest` last_dispatch 2026-04-27T07:54:52Z (97h+)
  - `heartbeat` last_dispatch 2026-04-30T20:14:40Z (12h+ — own 08:00 slot didn't fire)

## What needs to happen

1. Confirm GHA scheduler delivered no events to this repo between
   2026-04-30 22:13Z (last automatic tick — `evening-recap`) and now.
   Inspect Actions tab → Scheduled runs.
2. If the scheduler is silently dropping cron events, this is upstream
   GitHub infrastructure — there is no aeon-side fix. Workarounds:
   - Add an external watchdog (e.g. cron-job.org) that triggers
     `workflow_dispatch` on heartbeat every hour.
   - Use `repository_dispatch` from a more reliable scheduler.
3. If the scheduler is firing but the workflow file changed and no
   longer matches, audit the recent commits to `.github/workflows/`.
4. Once cron events resume, success_rate counters will burn down
   naturally for the affected skills. Skill-side code is healthy.

Pattern: GHA scheduler fires a tick, dispatches a batch of overdue skills, then goes silent for 2-6 hours. Today's tick history: silent through 08:52 (operator-invoked catch-up), 10:16 catch-up tick, 12:39 catch-up tick, then silent again. This is not a one-off — third consecutive day with multiple missed windows.

External-watchdog workaround is now blocking. Skills cannot recover faster than the next scheduler tick fires, and ticks are firing irregularly enough that 24h SLAs on daily skills are no longer dependable.
