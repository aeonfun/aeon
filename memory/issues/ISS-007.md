# ISS-007 — GitHub Actions fork cron throttle skips morning-review

## Summary

The morning-review chain did not fire at 2026-05-21T04:00Z. The previous chain
run was 2026-05-20T11:55Z, leaving a ~17-hour gap with zero scheduled-trigger
activity across all four workflows. Manual `gh workflow run chain-runner.yml -R
Azh1er/aeon -f chain=morning-review` (dispatched 2026-05-21T08:10Z) ran cleanly
end-to-end (9/9 steps), confirming the chain logic is intact and the throttle
is at the trigger layer only.

## Evidence

- `gh api '/repos/Azh1er/aeon/actions/runs?event=schedule&per_page=8'` shows the
  last schedule-triggered run was 2026-05-19T14:11:54Z (`tick: schedule + poll`).
  Every later run carries `event=workflow_dispatch` or `repository_dispatch`.
- All four workflows return `state: active` from
  `/repos/Azh1er/aeon/actions/workflows`.
- `/repos/Azh1er/aeon/actions/permissions` returns `enabled: true`.
- No queued or in-progress runs sitting in the actions queue.
- Main branch was pushed 2026-05-20T11:58Z — well inside any 60-day
  inactivity window.

## Impact

- High. Without manual dispatch, neither the morning-review chain (04:00 UTC)
  nor the heartbeat skill (08, 14, 20 UTC) fires. Daily briefs, macro reads,
  ops audits, and health checks all silently skip.
- This is the same failure shape that prompted moving the morning-review
  schedule from messages.yml's */5 poller onto chain-runner.yml directly.
  The fix moved the symptom from a high-frequency poller to a daily cron —
  the underlying fork throttle is unaddressed.
- Out of scope for the v4 perps-engine work but blocks v4's daily acceptance
  criterion (5 consecutive clean ledger lifecycles) unless the trigger is
  made reliable first.

## Hypotheses for why the throttle kicked in

- GitHub Actions has an undocumented per-fork rate limit on `schedule:` events
  separate from the 60-day inactivity rule. The trigger silently drops; no
  failed run is recorded.
- The high frequency of `workflow_dispatch` and `repository_dispatch` activity
  on this fork may interact with the scheduler's priority queue, deprioritizing
  cron triggers in favor of explicit-dispatch.
- Account-level Actions concurrency limits may park scheduled triggers when
  another workflow holds the per-account quota.

## Workarounds (immediate, low-effort)

1. **Manual morning dispatch.** `gh workflow run chain-runner.yml -R
   Azh1er/aeon -f chain=morning-review`. Used on 2026-05-21 — chain completed
   9/9 successfully.
2. **No-op commit to main.** Pushing any commit to the default branch
   sometimes re-arms scheduled triggers for a few days.
3. **Touch the Actions tab in the GitHub UI.** Anecdotally, opening the
   Actions page on the fork can nudge the scheduler.

## Structural fixes (longer-term, separate scope)

1. **External scheduler with PAT.** A cron job on the operator's machine (or
   any always-on host) calling `gh workflow run` via a personal access token
   sidesteps the fork-throttle entirely.
2. **Run the schedule on upstream (`aaronjmars/aeon`).** Upstream Actions are
   not subject to the same fork-trigger throttle. Requires either upstream
   merge of v4 changes or a routing mechanism that lets the fork inherit
   upstream's chain-runner schedule.
3. **Self-healing checker.** A check that runs early in `heartbeat` (or any
   reliably-triggered workflow) detects "no morning-review fired in last 24h"
   and dispatches one. Only helps if at least one trigger gets through.
4. **GitHub Issues comment trigger.** A tiny separate workflow on
   `issues.opened` or `issue_comment.created` that operator triggers from
   their phone — manual but more reliable than cron.

## Recommended next step

Adopt option 1 (external scheduler) as the structural fix. It removes the
dependency on GitHub's cron entirely and the operator already runs a
machine that can host a 1-line cron + 1-line gh command. Track separately
from v4 — different domain, different acceptance test.
