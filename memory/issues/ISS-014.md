## Reproduction

1. From the GHA sandbox, attempt the skill's primary XAI call as documented in
   step 1 of `skills/reply-maker/SKILL.md` — fails because `$XAI_API_KEY` expands
   to an empty string under sandboxed bash.
2. `node -e "console.log((process.env.XAI_API_KEY || '').length)"` → `0`.
3. `ls .xai-cache/` → no `reply-maker*.json` file (the prefetch case is missing).
4. WebSearch for `site:x.com Polymarket OR Kalshi prediction market April 27 2026`
   → returns a mix of older URLs and one within-window URL (`arkham/2048665135485735020`).
5. WebFetch on that within-window URL → HTTP 402.
6. Skip gate cannot be applied without verifiable text/engagement counts → skill
   correctly emits `REPLY_MAKER_EMPTY`.
7. **2026-04-28 17:17 UTC re-run:** WebSearch surfaced two within-window URLs
   directly (no WebFetch needed because the surrounding article snippets carry
   enough text to apply the gate). Both authored by `@Polymarket` and announcing
   the V2 cutover/launch — pure product-launch self-promo, fail the skip gate.
   Net actionable candidates: 0. Same `REPLY_MAKER_EMPTY` exit.

## What needs to happen

**Option A — add a `reply-maker)` case to `scripts/prefetch-xai.sh` (recommended,
mirrors the fix that closes ISS-001/002/012).**

Model the case on the existing `narrative-tracker` block in the same script:

```bash
reply-maker)
  # Pre-fetch reply-worthy original posts from last 6h
  FROM_DATE=$(date -u -d "6 hours ago" +%Y-%m-%dT%H:%M:%SZ 2>/dev/null \
    || date -u -v-6H +%Y-%m-%dT%H:%M:%SZ)
  TO_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ)

  if [ -n "$VAR" ]; then
    QUERY_DESC="for handle/topic/list: ${VAR}"
  else
    # Default to mission-aligned topics from MEMORY.md
    QUERY_DESC="about: Polymarket, Kalshi, Hyperliquid, prediction-market calibration, agentic finance"
  fi

  xai_search "reply-maker.json" \
    "Search X for the 12 most reply-worthy original posts (not retweets, not replies) ${QUERY_DESC} between ${FROM_DATE} and ${TO_DATE}. Reply-worthy = has a take, claim, question, or framing worth engaging — NOT pure self-promo, breaking news without analysis, or threads already past 500 replies. For each: @handle, full tweet text, tweet URL, posted_at ISO timestamp, like/reply/retweet counts." \
    "$FROM_DATE" "$TO_DATE"
  ;;
```

Then update `skills/reply-maker/SKILL.md` step 1 to read
`.xai-cache/reply-maker.json` first and fall through to the existing memory /
WebSearch chain only when the cache is missing.

**Option B — disable reply-maker until the prefetch case lands.** Remove the
cron entry to stop daily empty-notification spam.

**Option C — ship a Twitter-API-via-Apify or similar third-party path.** Higher
ongoing cost than option A; only worth it if reply-maker becomes load-bearing
for the LP-raise narrative.

## Notes

- Same shape as ISS-001 (vuln-scanner, 2026-04-25), ISS-002 (vibecoding-digest,
  2026-04-25), and ISS-012 (reddit-digest, 2026-04-26): a documented public-API
  path the skill calls is blocked at the edge, and the documented WebFetch
  fallback is also blocked. All four close on a single-line addition to a
  prefetch script (or, for the Reddit pair, a new `scripts/prefetch-reddit.sh`).
- The prefetch-xai.sh script template exists and works for five other tweet-fetching
  skills — adding reply-maker is the lowest-friction fix in this category.
- This is now the **second** sandbox-limitation issue tied directly to
  `scripts/prefetch-xai.sh` (after the ones that already use it). The pattern:
  any new tweet-sourcing skill needs its case added to prefetch-xai.sh as a
  prerequisite to working in production.
