title: messages.yml run-block interpolates ${{ github.event.action }} from repository_dispatch payload
status: open

detected_at: 2026-04-27
affected_skills:
  - workflow:.github/workflows/messages.yml
root_cause: |
  Line 578 reads `TYPE="${{ github.event.action }}"` directly inside a `run: |` shell
  block. `repository_dispatch` event_type is set by the dispatcher (any token with
  repo:write) and is interpolated into the shell context unescaped. A hostile
  event_type containing a single quote, `;`, or `$(...)` breaks out of the assignment
  and runs as the workflow shell user with full secret access (TELEGRAM_BOT_TOKEN,
  CLAUDE_CODE_OAUTH_TOKEN, ANTHROPIC_API_KEY, GH_TOKEN, etc.).

  Same incident class as the 2026-04-11 messages.yml audit — `inputs.message` and
  `inputs.source` were correctly rebound to `_INPUT_MESSAGE` / `_INPUT_SOURCE`
  env keys, but `github.event.action` and `github.event.client_payload.message`
  on line 577 were missed during that fix. The toJson() wrapper on line 577 does
  not protect against bash-quote breakout because JSON-encoded strings retain
  literal single quotes, which terminate the bash single-quoted string.

fix_pr: null

## Remediation
Rebind the github.event values to env keys, then read from the shell:

```yaml
- name: Extract message
  id: msg
  env:
    _INPUT_MESSAGE: ${{ inputs.message }}
    _INPUT_SOURCE: ${{ inputs.source }}
    _DISPATCH_ACTION: ${{ github.event.action }}
    _DISPATCH_PAYLOAD: ${{ toJson(github.event.client_payload.message) }}
  run: |
    if [ "$GITHUB_EVENT_NAME" = "repository_dispatch" ]; then
      MESSAGE=$(echo "$_DISPATCH_PAYLOAD" | jq -r '.')
      TYPE="$_DISPATCH_ACTION"
      ...
```

`$GITHUB_EVENT_NAME` is set by the runner and safe to read directly.
See `articles/workflow-security-audit-2026-04-11.md` for the canonical pattern.
