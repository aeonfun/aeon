# ISS-005 — Summary-blob artifact bug persists across 7 chain skills

## Summary

ISS-003 tracked chain skills writing the assistant `## Summary` block into
`.outputs/{skill}.md` instead of the locked-format artifact. ISS-004 escalated it
for perps-scan and shipped a structural fix (JSON intermediate + deterministic
postprocess render). Both issues are marked resolved as of 2026-05-19.

On the 2026-05-19 morning-review chain run audited by daily-ops-review, six of the
nine chain artifacts are Summary blobs:

- `.outputs/market-context-refresh.md`
- `.outputs/aixbt-pulse.md`
- `.outputs/narrative-tracker.md`
- `.outputs/monitor-runners.md`
- `.outputs/token-movers.md`
- `.outputs/morning-macro.md`

Only the three skills with a clean artifact this run are perps-scan (the v3
structured-render path — fix held), perps-brief, and token-call.

The 2026-05-19 daily-ops-review audit also found `.outputs/daily-ops-review.md`
itself landed as a Summary blob on a prior run that day — daily-ops-review has the
same bug, bringing the affected count to seven. The fourth ops-review run of
2026-05-19 regenerated its own artifact in the locked format and recorded the
expanded scope here.

The ISS-003/ISS-004 resolutions were scoped to perps-scan plus a prose guardrail
on narrative-tracker. narrative-tracker still produced a blob, and the other five
skills were never patched at all. The bug class is not resolved.

## Impact

- Low-to-medium. The blobs carry the headline data, so downstream consumers
  (perps-brief, morning-macro, daily-ops-review) still functioned this run.
- The locked-format payload is lost for any consumer that needs structure rather
  than prose, and for cross-day audit skills reading historical artifacts.
- Several blobs assert in their own text that they wrote the locked format — the
  artifact is silently wrong, not visibly empty.

## Recommended fix

Generalize the perps-scan v3 pattern to the remaining six chain skills, or fix the
underlying mechanism once for all of them:

- Skills write a structured intermediate (`.outputs/{skill}.data.json` or similar)
  and a deterministic `scripts/postprocess-{skill}.sh` / renderer produces the
  markdown after Claude finishes — the workflow already auto-loops `postprocess-*`.
- Or, if the chain-runner is overwriting `.outputs/{skill}.md` with the assistant's
  final message, fix the chain-runner to preserve the skill-written artifact.

The prose guardrail from ISS-003 is confirmed insufficient — narrative-tracker has
it and still failed.

## Workaround

A per-skill re-run regenerates the artifact, but ISS-004 already proved re-running
is not reliable (a deliberate corrective re-run reproduced the blob). daily-ops-review
is consume-only and does not re-run skills.


## Resolution (2026-05-20)

Root-cause fix shipped in `.github/workflows/aeon.yml` — the `Capture skill
output` step was UNCONDITIONALLY overwriting `.outputs/{skill}.md` by `cp
/tmp/skill-result.txt`, where `/tmp/skill-result.txt` is Claude's `.result` JSON
field (the chat response, which per CLAUDE.md convention ends with a `##
Summary` block). Every skill that wrote its locked-format artifact via the
Write tool had that artifact OBLITERATED by this copy. `perps-scan` and
`perps-brief` escaped only because their postprocess scripts ran AFTER capture
and rebuilt the artifact from a JSON intermediate.

Fix (one block in the workflow):

1. Touch `/tmp/aeon-claude-start` immediately before `claude -p` runs. This is
   the sentinel timestamp — anything in `.outputs/` newer than the sentinel was
   written by Claude this run.
2. Capture step now resolves in priority order: json-render pending → fresh
   skill-written artifact (mtime > sentinel) → fallback to `/tmp/skill-result.txt`
   → empty placeholder. The skill-written artifact is preserved when fresh.

This addresses the bug class for ALL affected skills (market-context-refresh,
aixbt-pulse, narrative-tracker, token-movers, token-call, morning-macro,
daily-ops-review) without per-skill JSON+render rewrites. The earlier
perps-scan/perps-brief JSON+render scripts continue to work — they overwrite
after Capture via postprocess, which still runs unchanged.

Verified by: tomorrow's morning-review chain run will be the first end-to-end
test. The mtime comparison is deterministic enough to expect zero false
preserves and zero false fallbacks.
