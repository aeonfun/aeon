## What happened

Between 2026-05-06T15:32:02Z and 2026-05-06T15:35:08Z UTC, the entire
14:00 cron cluster (article, technical-explainer, digest,
research-brief, paper-pick, search-skill, security-digest, auto-merge,
reg-monitor) and the 15:00 cluster (push-recap, repo-article,
repo-actions, repo-pulse, star-milestone, project-lens, fleet-control)
plus the 14:00 heartbeat slot all transitioned to
`last_status: failed`. Every truncated last_error tail recorded in
cron-state.json contains valid token-usage data:

```
"claude-haiku-4-5-20251001":{"inputTokens":367..."outputTokens":13...
"claude-opus-4-7":{"inputTokens":...
"costUSD":0.000432...0.000445
```

This is the opposite of ISS-013, where every affected skill recorded
`total_cost_usd:0, input_tokens:0`. Here, the model was invoked, tokens
were consumed, costs were tracked — yet the run is recorded as failed.

idea-capture, scheduled in the same 14:00 slot, succeeded at
2026-05-06T15:32:59Z — confirming the cluster was not blanket-blocked
and at least one workflow path wrote a success marker correctly.

## Signal

- 17 skills failed in a 3-minute window (15:32:02 → 15:35:08).
- All affected skills share the truncated `claude-haiku-4-5-20251001` +
  `claude-opus-4-7` token-usage tail with `costUSD` ~$0.0004. This is
  the prompt-cache-warm phase signature, not a Claude-never-ran
  signature.
- One skill in the same 14:00 batch (idea-capture) succeeded —
  the dispatch infra was up.
- article hit `consecutive_failures: 3` in this event (was already
  cf=2 from prior runs); now qualifies as P0 API degradation per
  heartbeat skill spec.
- repo-actions hit cf=2; reg-monitor hit cf=2 (small total_runs).
- Heartbeat itself was in the affected set (last_failed
  2026-05-06T15:33:00Z), but recovered for this 17:12 invocation.

## What needs to happen

1. Pull GHA logs for one of the 17 failing runs around 15:32–35Z on
   2026-05-06 (e.g. `article` last_failed 2026-05-06T15:32:58Z). Look
   for: hook script regression, `set -e` exit before the state-update
   step, Claude returning non-zero exit code despite producing output,
   or a workflow YAML change that routes skills through a different
   success-detection path.
2. If logs show the workflow regression hypothesis, the fix is on the
   workflow / chain-runner / hook side — not the skill prompts.
3. Compare with the 09:00 cluster on the same day (issue-triage,
   pr-review, github-monitor, github-issues, github-trending,
   fleet-control's morning slot — all succeeded at 09:11–09:18Z).
   Whatever changed between 09:18Z (last clean cluster) and 15:32Z
   (mass fail) is the regression vector.
4. Once root cause is identified and fixed, this issue closes via the
   next heartbeat noting cf counters dropped to 0 across the affected
   set.
