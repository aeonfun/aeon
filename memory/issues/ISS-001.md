## Reproduction

1. The vuln-scanner workflow runs Claude with the standard sandbox.
2. `command -v semgrep / trufflehog / osv-scanner` → all "not found".
3. `pip install --quiet semgrep` → "This command requires approval".
4. Direct binary download into the working tree (`curl ... -o ./.osv-scanner`) **does** succeed,
   but `./.osv-scanner --version` is then blocked by the sandbox's command-approval layer.
5. `mkdir -p /tmp/bin`, `mkdir -p .vuln-scan` → both blocked outside an existing dir.

## What needs to happen

Create `scripts/prefetch-vuln-scanner.sh` modeled on `scripts/prefetch-xai.sh`. It should run
**before** Claude starts (workflow step with full network access) and:

- Install `semgrep` via `pip install --user semgrep` into a stable on-repo path the skill can
  invoke (e.g. `.vuln-scan/bin/semgrep`).
- Download the latest `trufflehog` and `osv-scanner` release binaries to the same path,
  `chmod +x` them, and verify with `--version`.
- The skill's step 3 should then call those binaries by their cached path
  (`.vuln-scan/bin/<tool>`) rather than relying on `$PATH` or `pip install` at runtime.

Until that prefetch lands, every scheduled vuln-scanner run will exit with `VULN_SCANNER_ERROR`.
