# ISS-006 — monitor-runners artifact overwritten to `--help`

## Summary

On the 2026-05-20 morning-review chain run, monitor-runners executed cleanly and
the skill log records a 1.5kb v2-locked signal written to
`.outputs/monitor-runners.md`. The on-disk file holds the literal seven-byte
payload `--help\n`. The chain context passed to daily-ops-review (chain Step 4)
also shows the monitor-runners section as `--help`, so the overwrite occurred
between the skill's write and Step 4's read.

## Impact

- High. monitor-runners output is lost for this chain run. perps-brief and
  morning-macro consumed an empty section in place of the DEEP-LIQ verdict, and
  daily-ops-review cannot audit the runner-side signal.
- Different failure mode from ISS-005. The artifact is not the wrong format. It
  is replaced wholesale by an unrelated string.

## Hypotheses

- A `--help` invocation against the skill runner or notify script landed in the
  wrong place and shell-redirected over the artifact.
- A workflow step in `chain-runner.yml` or a postprocess script ran with an
  argument-parse path that wrote `--help` to stdout and clobbered the file.
- An aliased command in the sandbox printed its usage banner to the artifact path
  via a stray `>` redirect.

## Recommended next step

Inspect `chain-runner.yml` and `scripts/postprocess-*.sh` for any step that could
write to `.outputs/monitor-runners.md` after the skill exits. Check whether the
monitor-runners skill itself has a `--help` branch and whether a workflow step
shells the skill name with `--help` for documentation purposes.


## Resolution (2026-05-20) — DID NOT HOLD

Same root cause as ISS-005 — the Capture step was overwriting the skill's
Write-tool-created artifact. In this case, Claude's `.result` field was the
literal seven-byte string `--help` (likely because Claude tested `./notify
--help` to discover its options at the end of its run, and the chat response
captured that snippet as the final text). The Capture step then copied
`--help\n` over the 1.5kb locked-format signal monitor-runners had written to
`.outputs/monitor-runners.md`.

Fixed by the same one-block change in `.github/workflows/aeon.yml` documented
in ISS-005's resolution: the sentinel-mtime check now preserves the
skill-written artifact when it's fresh, so an unrelated string in
`/tmp/skill-result.txt` cannot clobber it.

This bug WAS a complete-replacement failure rather than a Summary-blob leak,
but it shared the same mechanism (overwrite of a skill-written artifact by a
fallback path that should only fire when the skill didn't write).

## Reopen (2026-05-20, later same day)

A later chain run on 2026-05-20 hit the same failure mode. `.outputs/monitor-runners.md`
on disk holds `--skill\n` (an eight-byte payload, different from the morning's
`--help\n` but same shape: a CLI snippet that should never be the artifact body).
The fix shipped earlier today did not prevent the second occurrence.

Hypotheses to investigate before re-closing:

- The sentinel-mtime check fires on the wrong file, or compares the wrong
  timestamps, so it does not protect this artifact.
- The fix landed in `.github/workflows/aeon.yml` but never reached the running
  workflow (cache, branch, or deploy-timing issue).
- A second overwrite path exists outside the Capture step that the sentinel
  check does not cover.
- Claude's `.result` field is again a stray CLI snippet (`--skill`), suggesting
  the model is sampling `--<word>` patterns from its tool-discovery sequence
  and the capture path still pulls that text into the artifact.

Recommended next step: dump the workflow run log for the failing chain and
trace every write to `.outputs/monitor-runners.md` from skill-exit to chain
Step 4 read. Add an early sentinel inside the skill itself that the workflow
can verify post-write.

## Recurrence (2026-05-20, later same day)

PR #27 merged at 07:34Z. The first monitor-runners run after that merge (at 07:35Z, run ID 26148366406) reproduced the bug — `.outputs/monitor-runners.md` came out as the eight-byte string `--skill\n` instead of the locked-format signal the skill log records as written.

Workflow log shows the Capture step took priority branch (1) — `Capture: used json-render pending file` — and copied `dashboard/outputs/.pending-monitor-runners.md` (which contained `--skill`) over the skill-written artifact.

Two distinct bugs interacting:

1. **`./notify` accepts flag-like arguments as MSG.** The script's arg parser only knows `--signal`. If Claude calls `./notify --skill <something>` or `./notify --help`, `$1` is `--skill` (or `--help`), the parser doesn't shift it, and MSG becomes the flag string itself. The json-render path then writes that string to `dashboard/outputs/.pending-${SKILL}.md` and the pending-notify path queues it for Discord.

2. **Capture priority was wrong.** PR #27 kept the json-render pending file as priority (1) and the fresh skill-written artifact as priority (2). When (1) carries garbage from bug #1, it wins over the correct artifact.

## Second fix (2026-05-20)

Two changes in `.github/workflows/aeon.yml`:

1. Notify script rejects flag-like MSG arguments: `case "$MSG" in --*) echo "notify: refusing flag-like MSG argument: '$MSG'..." >&2; exit 2 ;; esac`. The non-zero exit makes misuse visible in workflow logs instead of silently writing garbage to .pending- files.

2. Capture step priority swapped — fresh skill-written artifact is now (1), json-render pending file is (2). Only signal-only skills that route entirely through `./notify` (no Write-tool artifact) hit (2) now.

Combined, the failure modes from both today's recurrence AND yesterday's original `--help` incident are blocked at the source (notify rejects the flag) AND at the consumer (Capture preserves the correct artifact even if a stale or wrong pending file exists).
