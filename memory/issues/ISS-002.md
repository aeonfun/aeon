# ISS-002 — Coinglass coins-markets universe call fails on every variant

## Summary

`perps-scan` produces only the one-line "scan unavailable" artifact because the
Coinglass v4 `coins-markets` universe call fails in the prefetch. This is distinct
from the resolved ISS-001 (sandbox/auth): the prefetch infrastructure itself works
— `scripts/prefetch-coinglass.sh` executes, `COINGLASS_API_KEY` is present, and a
`manifest.json` is written. The failure is upstream, at the API call.

The prefetch was hardened by PR #9 (`coinglass-tier-fallback`) to step down through
five `coins-markets` variants (richest multi-exchange/per_page first, bare defaults
last). On the 2026-05-18 14:21 UTC run, **all five variants failed**:

```
exchange_list=Binance,Bybit,OKX&per_page=50;
exchange_list=Binance,OKX&per_page=50;
exchange_list=Binance,OKX&per_page=25;
exchange_list=Binance,OKX;
<defaults>
```

Because even the bare-defaults call fails, this is not a tier-gated *parameter*
problem — it is the endpoint, the key, or the account.

## History (all 2026-05-18)

- Runs 1–2: pre-prefetch / ISS-001 era (sandbox-auth failure — resolved).
- Run 3 (~13:58 UTC): prefetch infra worked, `coins_markets_ok: false`, no variant
  fallback yet. Logged as a possible one-off transient; follow-up said "monitor next
  run; if still false, escalate."
- Run 4 (~14:21 UTC): prefetch + PR #9 variant fallback in place, still
  `coins_markets_ok: false` across all five variants. **Recurred → escalated here.**

## Impact

- `perps-scan` has 0% success producing a real regime classification — every run
  emits only `Perps Regimes · <date> · scan unavailable, Coinglass API failed`.
- `perps-brief` (downstream consumer of `.outputs/perps-scan.md`) has no regime data.

## Recommended fix

1. **Close the diagnostic gap first.** `scripts/prefetch-coinglass.sh` only writes a
   `## Coinglass Prefetch Error` log block for HTTP 401/403. Extend it to also
   persist, for the final failed variant: the HTTP status code, the curl exit code,
   and the Coinglass `code` + `msg` from the response body. Without this the root
   cause is unknowable from automated runs.
2. **Then verify the account.** Once the real error is visible, check whether the
   Coinglass plan covers `/api/futures/coins-markets` (an "Upgrade plan" `msg`
   confirms a tier-gate), whether `COINGLASS_API_KEY` is valid/unexpired, or whether
   Startup-tier rate caps are being hit (HTTP 429 surviving the single retry).
3. If the endpoint is permanently tier-gated, either upgrade the plan or migrate
   `perps-scan` to a Coinglass endpoint the Startup tier does cover (re-categorize
   this issue to `permanent-limitation` or `api-change` once confirmed).

Steps 1 and 3 require a branch + PR; step 2 needs operator access to the Coinglass
account dashboard.
